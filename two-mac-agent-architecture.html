Operator reference · always-on agent architecture

The two-Mac agent architecture, hardened

An always-on agent on a dedicated Mac Mini, with focused coding on a MacBook, is the default pattern for serious operators. The separation keeps the interactive coding environment fast and safe while the agent gets a stable home for long-horizon tasks, cross-project memory, and event reactions while the laptop sleeps. The piece almost everyone is missing is deterministic, read-only context sync. Install one skill first, harden three boundaries this week, and the topology becomes a force multiplier instead of a delayed-reaction robot.

The thesis in one line The Mini is your asynchronous operating-system layer. The MacBook is your precision instrument. The private GitHub mirror is not a point of fragility: it is the deliberate interface that enforces clean boundaries between them.

01The core idea: separate the async layer from the instrument

Two machines, one one-way data spine. Code flows MacBook to mirror to Mini. Memory and logs flow Mini back to a different repo under a different token. The control path is human-initiated only.

Three boxes, one spine. The Mini holds a read-only view of code and writes only to a separate notes repo under a write-scoped mirror token. The dashed control path is the only way the agent acts on demand, and it is never the agent reaching out first.

02Install this skill first: Repo Watcher and Diff Summariser

It removes the stale-context root cause that silently degrades every other skill. Without it, every downstream capability works from a fading photograph of your codebase.

The skill watches the repositories you actively work on, detects new commits on tracked branches, fetches the diff, and produces a structured summary of what changed and why. The summary lands in the agent project-memory store keyed by repo:branch:date, so every other skill (code review, stand-up notes, backlog grooming, cost attribution) can query a current world model instead of re-learning the codebase each morning.

Why it compounds fastest

  • Removes the stale-context root cause that quietly degrades everything downstream.
  • Turns a passive tool into an active participant that updates its own model as you ship.
  • Low risk by design: read-only, no writes on the coding repo.
  • Generates clean cost-attribution data you can feed straight into the dashboard.
Setup notes (high confidence, current community patterns)
  1. Create a fine-grained personal access token scoped contents: read on the target repos. Store it in the macOS Keychain, not a dotfile.
  2. Install the official GitHub MCP server in the agent MCP config. A gh CLI shell skill works, but MCP is preferred for structured data.
  3. Skill definition includes: poll interval (start at 120 s), model preference (cheap-fast tier for summaries, reasoning tier only for complex architectural diffs), deduplication by commit SHA, max summary length.
  4. Output to a memory document keyed by repo:branch:date.
  5. Add a heartbeat: if no new commits for 12 h, log a note so you know the watcher is not frozen.
Success criteria and required logging

Success: every push appears in memory within 2 to 5 minutes; dashboard shows one summarisation call per push at a few cents on the cheap tier; a manual query "what did I change in repo X today" returns a coherent summary.

Must log: repo, branch, commit range, timestamp; model used, tokens in and out, cost; the summary text; status (success, rate-limited, empty diff, error).

Pick the right first skill for your profile

Stick with the Repo Watcher. It keeps your personal context continuous and stops you re-explaining the codebase to the agent every morning. Highest leverage, lowest risk.
Start with an Environment Audit and "doctor" wrapper. It checks every repo, SSH key, MCP server health, API key expiry, and disk space across the fleet. You cannot afford to discover breakage during a client deadline.
Start with an inbox / Telegram command-router skill. Send commands from your phone, receive structured reports. The agent becomes a remote-control panel for the homelab and code pipelines.

launchd PATH / env mismatch
launchd does not source .zshrc, so git, gh, node, and Python are often absent. Fix: set EnvironmentVariables in the plist, or run the agent through a wrapper script that exports an explicit PATH before launch.
Stale code context
The local clone is not updated often, or a fetch fails silently. Fix: the Repo Watcher with commit-SHA dedup, plus a health check that verifies last fetch time is under one hour.
API key leakage via env files
.env files get copied between machines via Git, AirDrop, or unencrypted backups, and one commit exposes them. Fix: store every secret in the macOS Keychain (read via /usr/bin/security) or a secrets manager. .env is for local dev only and is never committed.
Accidental write to wrong repo
A write-scoped token plus an incorrect remote URL in a skill lets the agent push where it should not. Fix: give the agent a read-only token for code repos, a separate dedicated token for the config mirror, and set git remote explicitly in every skill.
SSH agent forwarding exposure
When the Mini SSHes to the MacBook to trigger a coding task, agent forwarding can expose your keys onward. Fix: never forward the agent for automated connections. Create a dedicated key pair with no passphrase, restricted by a forced command= in authorized_keys.
Model cost runaway
A skill retries on error without backoff, or a poll interval is set too low. Fix: exponential backoff with jitter in every polling skill, a global daily spend cap, and a watchdog that hard-kills the launchd job if the cap is exceeded.
MCP server duplication / port conflict
Both machines try to run the same MCP server on the same port across an SSH tunnel. Fix: assign distinct MCP responsibilities per machine, prefer Unix sockets or localhost-only ports, and document port assignments in the config mirror.
macOS sleep / network instability
The Mini still sleeps if Energy Saver is wrong, or Wi-Fi drops. Fix: sudo pmset -a sleep 0, wired Ethernet, and a health-check skill that pings outbound and restarts networking if it fails.

04A day on the two-Mac system

The agent works the gaps. You work the focus. The handoffs are explicit and human-paced.

Recovery process (target: working again in under 10 minutes)
  1. SSH in, check launchctl list | grep <agent>.
  2. Run the recovery script from the mirror: git clone <mirror> /tmp/recovery && ./install.sh.
  3. Restore secrets from Keychain or the secrets manager into the fresh environment.
  4. Re-enable the launchd job. The Repo Watcher rebuilds memory from the last 24 h of commits.
  5. If the database is corrupt, delete it and let the agent rebuild from the mirror. Memory loss is limited to unsynchronised local notes.

05Security baseline

Do the left column first. The right column is the list of things that buy you future pain.

Do this first

  • Dedicated non-admin agent macOS user account.
  • All keys in Keychain, read into env at launch. Zero plaintext.
  • Two tokens: read-only on all code repos; write-only on the config mirror.
  • Dedicated SSH key, no passphrase, locked to one forced command.
  • App firewall on; dashboards bound to 127.0.0.1; private networking only.
  • Approval gate on any write outside the agent data directory.
  • Separate profiles or tokens per client / work / personal context.

Do not, unless you enjoy pain

  • Commit .env or any secret to Git, even a private repo.
  • Run the agent as your own user account.
  • Grant a full repo or admin scope token.
  • Reuse your personal SSH key for the agent.
  • Install community skills without code review (a skill can curl | bash).
  • Disable the cost cap because nothing bad has happened yet.

06Cost-control system

A polling loop or retry storm can burn a night of credits. The dashboard exists to make that visible. The watchdog exists to make it impossible.

TriggerThresholdAction

Single callover your per-call ceilingFlag for inspection
Retry rateover 10% in an hourFlag runaway loop

Model routing heuristics

  • Diff summaries, commit classification, log parsing: the cheap-fast tier (current Haiku). Volume work, no reasoning needed.
  • Complex code review, architectural analysis, deep memory writes: the reasoning tier (current Sonnet).
  • Non-critical, high-volume triage (which notifications to ignore): a local model. Free, always available, latency acceptable.
  • The frontier tier (current Opus): not for automated background tasks without an unlimited budget and a strong justification.
  • Spend optimisations: only call the model when commits actually change; batch all commits from a 30-minute window into a single summary; lean on the built-in spend cap.

07The config mirror as a recovery artifact

The mirror is not a backup folder. It is the single place a fresh agent instance can be fully reconstructed from. Version templates and declarations, never transient state.

Commit declarations and templates. Keep secrets, logs, and the database out. 
agent-mirror/
├── skills/            # declarative YAML, no secrets inline
│   ├── repo-watcher.yaml
│   ├── code-review.yaml
│   └── cost-summary.yaml

Field note to share if someone asks

A week into the two-Mac setup it is already worth it, with teeth in the first few days. The skill to install first is a repo watcher and diff summariser: point it at your main work repos, let it write a short plain-language summary into memory on every push, and you never have to explain "what I did this morning" again. Without it, every other skill is half-blind.

Real bites: launchd does not share your terminal PATH, so half the skills failed silently until they ran through a wrapper that set PATH explicitly. A write-scoped main token let the agent try to push a "fix" to a repo it had no business touching (no harm, it was a test repo, then immediately split to read-only code and a separate mirror token). The cost dashboard caught a retry loop that chewed real money in an hour: set hard caps, do not skip that.

Treat the config mirror as a recovery artifact from day one, keep secrets in the Keychain, and never let the Mini SSH into your coding machine without a restricted key and an approval step. Sensible setup once the sync and security boundaries are right.

Glossary
Repo Watcher
The read-only skill that turns each push into a memory entry, keeping every other skill context-aware.
Config mirror
A private repo of declarative config and templates. The single place a fresh agent can be rebuilt from.
launchd
macOS service manager. Runs the agent as a daemon, but does not inherit your shell PATH.
MCP server
A structured tool endpoint (GitHub, filesystem, memory) the agent calls. Assign one owner machine per server.
Spend cap
A hard daily ceiling. A watchdog unloads the launchd job if it is breached.
Approval gate
A human checkpoint before any write outside the agent data directory.
Recovery artifact
A clone-plus-secrets path back to a working copy in under two minutes.
