Running ten client stores through Claude without the disconnect dance

The verdict

The consumer Claude Shopify connector binds one store per account session. Changing stores means a full disconnect and a fresh OAuth handshake every time, so it is a non-starter for an agency juggling ten clients. The workflow you actually want, connect once and switch fluidly, already exists, just not in the web chat. It lives at the Claude Code plus Shopify API layer: credentials are stored once per client, and switching becomes a sub-second pointer change (a CLI switch or an MCP set_active_store call) rather than a re-authentication. The production-grade shape is a multi-store-aware MCP server holding per-client scoped tokens, with Claude Code as the reasoning layer and a hard human gate in front of every write.

Why fluid switching is possible at all

The trick is not a magic button. It is separating where the keys live from which lock is currently selected.

The consumer connector fuses those two things into one OAuth session, so the only way to point at a different store is to tear the session down and rebuild it. The CLI and MCP paths split them apart. Credentials sit on disk or in a vault, persistently, one set per client. The active-store pointer is a tiny piece of mutable state that any tool call can flip in well under a second. Connect once becomes a one-time setup; switch fluidly becomes a pointer write.

Six architectures, ranked for agency reality

Expand each. The badge is the bottom line for a ten-client book.
A · Consumer connector onlyNon-starter

One OAuth connection at a time in the web or desktop app. Moving from Store A to Store B means disconnecting A, reconnecting B, and sitting through the OAuth flow on every switch. Acceptable for a founder poking at a single store. Disqualified the moment a second client exists.

B · Claude Code + Shopify CLIStrong for dev work
How it works
  • Authenticate each client store once with shopify login --store clientA.myshopify.com. The CLI persists credentials locally, one set per store.
  • Inside a Claude Code session, run shopify switch --store clientA.myshopify.com to flip the active store, or set SHOPIFY_FLAG_STORE per call.
  • Switching is effectively instant. There is no re-authentication.
Limits
  • Best for theme, app, and CLI development, plus scripted bulk operations. It is not a conversational analytics surface inside web chat.
  • All tokens live on one machine. Without a validation step, a session can target the wrong store if the env var drifts. A write confirmation gate is mandatory.
C · Official Shopify MCP serverBest conversational fit, verify scoping

Expected to support a config file mapping store handles to tokens, with a switch tool such as set_active_store. Once set, products, orders, and analytics tools act on the chosen store, so a request like "show yesterday's sales for ClientX" can auto-switch and answer in one turn.

The open risk: if the server shares global state instead of scoping per conversation turn, a single session can silently query the wrong store after a switch. This must be proven with a two-store dry-run before any client data touches it.

D · Custom agency MCP serverRecommended long-term

A thin orchestration service exposes tools that take an explicit store parameter. It holds a secure vault of per-client Admin API tokens and routes each request to the correct store's GraphQL endpoint. You own audit logging, rate limiting, scope enforcement, and a propose-then-confirm gate on every mutation. Development cost is real and pays back fast past ten stores.

E · One workspace per clientIsolation at a cost

Separate project folders, each with its own tools and credentials. Leakage across clients becomes structurally impossible, which suits regulated work or staff who must be firewalled to one account. The price is ten environments, duplicated config, no cross-client aggregate view, and constant manual context switching.

F · Third-party connector as primaryReporting only

Pre-built vendor integrations can bundle multi-store support on their backend, useful for scheduled emailed reports or dashboard exports. As a primary operational stack they fail the agency test: limited scope control, client data traversing a third-party cloud, weak real-time behavior, and no custom tool definitions over raw store data. Liability is too high to make this the spine.

The build worth standardizing on

Claude Code reasons. A multi-store MCP server (official, wrapped, or custom) routes. A vault holds scoped per-client keys. Nothing mutates without a human.

The security model that keeps ten clients separate

Three concerns, addressed concretely. No shared tokens, no implicit active store, no unconfirmed write.

Provision one Shopify custom app per client store, each with the minimum scopes the work actually needs.

  • One token, one store, always. Never reuse a single Admin API token across stores, not even for a quick test. Cross-store reuse erases the audit trail and widens the blast radius to your entire book.
  • Tokens never sit in Claude's plaintext context. Keep them in a gitignored .env, a secret manager, or the MCP server's encrypted config. Claude Code reads them via environment, it does not hold them.
  • Partner access for onboarding, custom apps for the long run. Collaborator tokens are fine to bootstrap a new client. Durable API access should be a scoped custom app so every call is attributable.

Each failure mode has a specific countermeasure. Implement all of them, not the convenient subset.

  • Cross-store leakage: every tool requires an explicit store parameter. No tool ever assumes a global active store.
  • Wrong-store write: before any mutation, Claude echoes the target store name and the exact change, then stops for human approval.
  • Token sprawl: catalog every token in a central secure inventory and rotate on a fixed 90-day cycle.
  • Overbroad scopes: reporting uses read-only tokens with zero write scopes; write tokens are limited to the resources the task touches.
  • Hallucinated destructive action: the operator system prompt forbids any delete or bulk price change without a second, separate, explicit confirmation.

Standing controls that apply to every client, every session.

  • Read-only analytics channel: all reporting flows through tokens that physically cannot write.
  • Dev-store staging: every automation runs against a development store before it ever touches a client.
  • Isolated workspace for high-risk ops: bulk inventory or price work gets its own project, its own token, and a peer-review step.
  • Approval gates on automation: recurring reports may not write; mutations require an authorized teammate's explicit sign-off, ideally via a Slack approval before the run proceeds.
  • Audit everything: timestamp, store, operator, tool, and result summary for every call, reviewed weekly.

Operator setup, copy ready

Four artifacts you can lift directly. Replace aliases and tokens with your own. These are the spine of the minimum viable ten-store setup.
# System prompt: agency Shopify operator

You are an agency Shopify operator working across multiple
client stores. You never assume an active store.

1. Every tool call carries an explicit store alias.
2. Before any read, state which store you are querying.
3. Before any write (create, update, delete, bulk edit,
   price change, inventory adjustment): output a plain
   diff of exactly what changes, name the target store,
   then stop. Wait for the human to reply
   "APPROVE <store-alias>".
4. Deletions and bulk price or inventory changes require
   a second, separate confirmation: the human must type
   "I CONFIRM DELETE" or "I CONFIRM BULK".
   One approval is never enough for destructive actions.
5. If unsure which aliases exist, call list_stores first.
6. Log every store switch and every mutation: timestamp,
   store, tool, one-line summary.
7. On any token or scope error, stop and report. Never
   retry the same action against a different store.
# .env  (gitignored, rotate tokens every 90 days)
# One custom app per store. Least scope per task.

CLIENT_A_STORE=client-a.myshopify.com
CLIENT_A_TOKEN=shpat_xxxxxxxxxxxxxxxxxxxxxxxxxxxx
CLIENT_A_SCOPES=read_products,write_products,read_orders

CLIENT_B_STORE=client-b.myshopify.com
CLIENT_B_TOKEN=shpat_xxxxxxxxxxxxxxxxxxxxxxxxxxxx
# reporting-only tier: zero write scopes
CLIENT_B_SCOPES=read_products,read_orders,read_analytics

# Keep read-only reporting tokens in their own block
# so a write task can never grab one by accident.
{
  "stores": {
    "clientA": {
      "shop":  "client-a.myshopify.com",
      "token": "${CLIENT_A_TOKEN}",
      "mode":  "write"
    },
    "clientB": {
      "shop":  "client-b.myshopify.com",
      "token": "${CLIENT_B_TOKEN}",
      "mode":  "read"
    }
  },
  "defaultStore": null,
  "requireExplicitStore": true,
  "confirmBeforeMutation": true,
  "auditLog": "./logs/store-actions.jsonl"
}
# Authenticate each client store once.
# Credentials persist on disk, no re-auth on switch.
shopify login --store client-a.myshopify.com
shopify login --store client-b.myshopify.com

# Inside Claude Code, flip the active store (sub-second):
shopify switch --store client-a.myshopify.com

# Or scope a single call without a global switch:
SHOPIFY_FLAG_STORE=client-a.myshopify.com shopify <cmd>

# Always confirm the active store before operating:
shopify config show

Prove the server scopes store state per conversation turn, not globally, so a post-switch call cannot pollute another store.
Confirm with the platform vendor whether any multi-store consumer connector or beta is on the roadmap this year.
Audit CLI token storage on disk: verify nothing is cached in plaintext without explicit consent.

The rollout

This week
  • Treat the consumer connector as out of scope for agency work. It will not be multi-store soon.
  • Stand up a proof of concept: Claude Code plus the official MCP server, two test stores, switch and read verified.
  • Define credential storage and a least-privilege token plan. Mint scoped custom-app tokens per store.
Next 30 to 60 days
  • Build a thin custom MCP server (or wrap the official one) that enforces store confirmation and logs every action.
  • Fold it into one shared Claude Code project with written operating procedures for the team.
  • Make human approval mandatory for every write. No autonomous deletes or bulk price changes, ever.
Do not
  • Reuse one Admin API token across stores, even for a quick test.
  • Route all store data through a third-party cloud as the primary stack.
  • Assume the in-chat connector will gain multi-store overnight. Plan for the CLI and API stack as permanent.
Revisit quarterly
  • Watch the Shopify AI Toolkit changelog and the platform connector announcements.
  • If an official multi-store UI ships, it may absorb casual conversational tasks.
  • Keep the CLI and MCP safety net regardless: production mutations still need the gate.

Connect once, switch fluidly is not a feature you are waiting for. It is an architecture you assemble: Claude Code as the reasoning layer, an API-native multi-store router, isolated per-client keys, and a human standing in front of every write. Build for that, and the ten-store operating system is already in your hands.
